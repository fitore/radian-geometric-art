03 — Workflow

A feature from idea to shipped

Use tstack for planning and decisions. Use gstack for execution and verification. They compose — don't blur them.

01
Planning

/team — Pressure-test the idea

Describe what you want to build. PM scans competitors and sets requirements. PE audits existing systems and flags reserved contracts. UX maps the full journey and finds the real gap. Synthesis names the real tensions.

02
Planning

Review and approve the CLAUDE.md UPDATE block

The PE produces this automatically. Review it. If you approve the decisions and locked assumptions, paste it into your project's CLAUDE.md or AGENTS.md. This is how context compounds across sessions.

03
Execution

/team — Lock the spec, then build

Prefix with "Execution Mode —" to tell Claude the spec is locked. PE leads implementation. PM confirms scope hasn't crept. UX flags any journey breaks introduced by the implementation. No re-ideation in this mode.

04
gstack /review

Paranoid staff engineer review

Green tests don't mean it's safe. gstack's /review looks for N+1 queries, race conditions, trust boundary violations, and the bugs that survive CI. Run this before /ship.

05
gstack /ship

Sync main, run tests, push, open PR

Release machine mode. Spec is decided, review is done, branch is ready. This is not for deciding what to build — it's for landing the plane.

06
gstack /browse

QA pass — give Claude eyes

Claude Code navigates to staging, logs in, clicks through every changed page, takes screenshots, checks the console. The full QA loop runs in ~60 seconds. You don't open a browser.

07
gstack /retro

End of week: commit history, patterns, velocity

Saves a JSON snapshot for trend tracking. Run /retro compare to see this week vs last week. Useful for actually knowing if the system is compounding.

05 — The Overlay

When to add /regulated

Regulated Industry Overlay

This skill doesn't replace the team — it adds a compliance constraint layer that every role runs their output through. Invoke it alongside /team whenever the work touches regulated contexts.

🗂️ PM lens

Compliance as competitive moat. Regulatory surface scan. Auditor-useful evidence, not just internal checkboxes. Explicit human sign-off points mapped before requirements are written.

⚙️ PE lens

Audit-first architecture. Machine-readable log of every AI-assisted decision from day one. Confidence score shown with explicit provenance. Human approval logged as a separate event from AI recommendation.

🎨 UX lens

Two critical gaps: the post-proposal/pre-decision moment (where AI hands off to human judgment) and the downstream audit review moment (auditors as users). Both must be designed explicitly.

No machine-readable audit trail of AI-assisted decisions — this is a blocker, not a backlog item
Conflating AI confidence score with human approval — they are different events, log them separately
Fully automated consequential decisions with no human override path
Audit data requiring significant transformation before an auditor can read it

07 — Working patterns

Tips from how this was designed

🔀

Don't blur Planning and Execution Mode

gstack's key insight: planning is not review, review is not shipping. Your parallel: Planning Mode is not Execution Mode. Explicitly declare which mode you're in at the start of every session.

⚡

The synthesis block is the most valuable output

The "where we conflict" section is what you can't get from a single-role assistant. A PM differentiator that takes 3 months and solves the wrong journey moment is a real conflict — the synthesis names it.

🔒

Add /regulated early, not after the fact

Retrofitting an audit trail is expensive. If there's any chance the work touches regulated contexts, run /team /regulated in the first planning session, not after the architecture is locked.

📌

Treat reserved contracts as hard constraints

The populateForm() hook pattern — reserve the interface, don't implement it. The PE will flag any approach that violates it. This is how v2 integrations stay clean even when built months later.
